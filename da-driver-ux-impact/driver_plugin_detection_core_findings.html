核心發現與 BQ 證據

外掛司機識別 — 行為信號、高風險場景、數據支持、定義

BQ / Ops / T&S

核心發現

接單速度無法有效識別外掛。真正有區分度的是一整組可量化的訂單操縱行為。

可分為兩類：

1. 直接操縱證據

放棄後重搶同單、重搶後附加費上升

2. 輔助行為輪廓

高 release_rate、Grace Drop / Driver cancel 異常、棄低搶高、最終 net_impact_hkd < 0

可量化信號如下：

  • Release rate 異常高（超出同等級 × 同車種司機 p90）
  • Release 後重搶同一筆訂單
  • 重搶後成交加成明顯高於首次
  • Grace Drop / 司機主動取消偏高，且集中於低價單
  • 高價單接受率高、低價單選擇性放棄
  • 最終對平台造成實際虧損（net_impact_hkd < 0）
  • 到達時間 / 距離只可作訂單難度控制變數，不可直接當作外掛證據
外掛司機的異常行為為什麼是問題怎麼在 BQ 看
Release 偏高搶到單後不是要完成，而是先放掉來推高加成release_rate 高於同等級 × 同車種的 p90
Release 後重搶同一筆單最像「搶單 → 放單 → 再搶單」的套利循環同一 driver_id 放棄後，再次接觸同一 order_request_id（最強信號）
加成升高後才成交Release 讓車資變高，司機再回頭吃高價surcharge_delta > 0；1 次約 +8%，2 次以上約 +15%
Grace Drop / 司機取消偏高可能是在避開正式 release_rate，或選擇性放掉低價單cd=21 = 接單後極短時間內放棄（快接快甩）；cd=3 AND actor_type='Driver' = 由司機主動發起取消
高價單 / 機場單占比偏高，低價單放棄偏多有外掛輔助時，更容易挑高報酬的單來接機場單接受率 vs 低 GTV 訂單的 release / cancel rate
造成用戶重下單 / 等待更久 / 流失這才是平台真正要處理的商業傷害is_reordered、UX harm、net_impact_hkd
事件碼白話翻譯：
  • cd=21（Grace Drop）= 司機接單後在極短時間內放棄，較接近「接咗即刻甩手」的特殊棄單；在 OD 單近 30 天實查中，接觸到 Grace Drop 的時間差 p50 = 3 秒、p90 = 6 秒
  • cd=3 AND actor_type='Driver' = 這宗取消是由司機主動發起，不是乘客取消，也不是系統自動取消
  • 以上兩者都屬事件層級（event-level）記錄，可細到單張訂單、單一司機、單次事件時間點；不是只到日彙總

高風險訂單組合：難派且易受外掛操縱的場景

BQ 已確認。以下分類幫助 Ops 決定在哪裡重點監控、哪裡可以暫緩。

本節按三類場景分類：

  1. 優先監控場景
  2. 自然市場行為場景
  3. 供給不足場景

用途是排序覆核優先順序，避免把人工覆核資源放在錯的場景。

① 高風險組合

以下三個條件同時出現時，最容易出現「放棄 → 等加價 → 再接」：

條件說明BQ 信號
高加成貨物機場趕班 avg HKD 45 / 搬遷 avg HKD 17；放棄率最高request_remarks 關鍵字分類；機場單 destination_location_region
稀疏競爭池同一筆單曾被分配 ≥ 4 次（count_assignations ≥ 4）；bot 知道訂單難找人，放棄可有效推高加成order_master_table_hk.count_assignations
傍晚 / 深夜時段供給緊張，放棄後加成升幅更大EXTRACT(HOUR FROM created_at)

加上 bot 本身的確認信號：release_rate 高 AND 同單重接率高

共同特徵為：訂單附加費高、競爭池稀疏、時段位於供給最緊區間。司機放棄一輪後，更有機會等到平台調高加價，再接回同一張單。

⚠️ 寵物 / 急件醫療 已移出高風險清單：BQ 確認加成極低（HKD 5–11），放棄率低於平均，bot 無經濟誘因針對——Ops 無需對這兩類貨物特別加強監控。

② 最容易被推價的區間

加成 $31–60 × 中距離 GTV $200–350 × 傍晚 18:00–20:00

此組合最容易出現「棄單 → 等加成升 → 重搶」：

  • 加成已夠高，值得操縱；但未到 $61+，平台動態加成規則仍有上升空間
  • 中距離 GTV 代表用戶議價能力低，容易被迫接受更高車費
  • 傍晚是供需缺口最大的時段

在這個區間內，若同時看到 release_rate 高 + 同單重接，覆核優先級應提高。

該區間同時具備高收益、可操作性高，以及對用戶傷害較大的三項特徵。

③ 自然搶單場景

加成 $61–100 × 早晨 06:00–08:00

此區間司機自然搶單意欲高，不屬於外掛信號。若偵測到高 release rate，需先排除「早晨高加成單被合法搶走後放棄」的正常行為。

高加成與高搶單意願本身不代表外掛。若缺乏重搶同單、加價後再接或其他操縱證據，不應僅因訂單熱門而提高制裁優先級。

④ 供給不足場景

加成 $1–15 × 午間 13:00–15:00 × 中距離 GTV $200–350

此類訂單的高棄單率主要由供給不足造成，並非 bot 操縱。誤把這類訂單的高棄單率歸因於外掛，會污染 Baseline Detection Model 的 p90 門檻。

Ops 覆核時若發現候選司機主要棄的是這類訂單，應降低信心分數，不建議制裁。

若主要問題來自供給而非操縱，對應措施應為供給修復、加成規則優化或派單策略調整，而非優先處理司機名單；否則會把供給問題誤判為紀律問題。

解讀：

  • 短期：被 release 的單平均加成更高（+$1.8）、佣金更高——單從單筆角度，平台可能多賺一點

結論：客人加錢，平台短期未必吃虧；但只要放棄行為讓完成率下降、補償變多、客人之後不再回來，整體平台淨影響會由「短期多賺」變成「中長期少賺甚至貼錢」。

⑤ Grace Drop 低估問題：實際棄單行為可能被嚴重低估

30 天實查（2026-06-15，HK TRANSPORT，raw source_ts，Driver）：

棄單類型說明事件數涉及司機數目前計入棄單率？
正式 Release（cd=4）司機主動放棄已接訂單91,356 宗—✅
Grace Drop（cd=21）司機在接單後極短時間內放棄，系統視作特殊取消436,366 宗5,376 位❌

Grace Drop 在近 30 天的事件量約為正式 Release 的 4.8 倍，而且目前完全不計入棄單率；若只看正式 Release，實際棄單行為會被明顯低估。

計法要分開看：
  • 事件量計法：看近 30 / 90 / 180 天累積有多少宗 cd=21，用來判斷 Grace Drop 是否屬高量體事件
  • 每單時間差計法：看單一訂單從接單到 Grace Drop 相隔幾多秒 / 分鐘，用來判斷它是否屬「快接快甩」

OD 單每單時間差（近 30 天，raw source_ts，On Demand，Driver）：

  • 接觸 → Grace Drop：p50 = 3 秒、p90 = 6 秒、p95 = 7 秒
  • 接觸 → 正式 Release：p50 = 37 秒、p90 = 204 秒（約 3.4 分鐘）、p95 = 446 秒（約 7.4 分鐘）

白話理解：Grace Drop 更接近「接咗即刻甩手」；正式 Release 則較多發生在接單後數十秒至數分鐘。

Product 需決定是否將 Grace Drop 納入計算，偵測門檻才能被正確校準。

⑤ 到達時間與距離：只適合做控制變數，不適合直接做外掛信號

BQ 實查（2026-06-12，HK TRANSPORT 預約單，近 30 天）：

  • gogox-analytics-prod.hk_db.orders.driver_arrived_at 可透過 order_request_id + driver_id join 回訂單
  • 已完成訂單中，driver_arrived_at 覆蓋率 100%
  • order_master_table_hk.waypoint_straight_distance_km 覆蓋率 98.3%
  • 但 request_pickup_time → driver_arrived_at 分佈很廣：p05 = −24 分鐘、p50 = 9 分鐘、p95 = 95 分鐘

上述分佈反映的是訂單本身的難度，例如預約單提早接單、特定區域供應不足、中長途訂單本來就較難派，不是外掛操縱的直接證據。

高棄單司機（top 10% release）與低棄單司機（bottom 50% release）比較：

群組request_pickup_time → driver_arrived_at 中位數waypoint_straight_distance_km 中位數
bottom 50% release6 分鐘10.5 km
top 10% release20 分鐘17.1 km

判讀：高棄單司機接到的訂單，平均距離更長、難度更高、到達更慢。可以用來解釋「這張單本身比較難派」，但不能單獨作為外掛證據。

BQ 欄位與計算公式（控制變數用途，不作制裁依據）：

-- 預約時間到司機到達的時間差（訂單難度 proxy）
-- ⚠️ 起點是 request_pickup_time（預約時間），不是接單時間
pickup_to_arrive_min
= DATETIME_DIFF(
    DATETIME(orders.driver_arrived_at, 'Asia/Hong_Kong'),
    order_master_table_hk.request_pickup_time,
    MINUTE
  )

-- 直線距離 proxy（非實際行駛公里）
distance_proxy_km
= order_master_table_hk.waypoint_straight_distance_km
⚠️ 限制：
  1. driver_arrived_at 適合觀察履約節奏，不適合直接識別外掛
  2. waypoint_straight_distance_km 只是直線距離估算，不是實際行駛公里數
  3. 若要使用，應放在 booking_type × hour × pickup_region × distance_bucket 的分層比較中

外掛司機定義

本專案所指的「外掛司機」，不是接單速度快的司機，而是疑似透過第三方 bot 反覆執行以下操縱循環的司機：

搶單 → 故意 Release → 等待平台提高動態加成 → 重搶同一筆單 → 以更高費用成交

識別依據是訂單操縱行為，而非接單速度。

進入制裁候選須三個主條件同時成立：

  1. Release rate 超出同等級 × 同車種司機 p90
  2. 至少一項惡意信號：重搶同單，或 Release 後加成升高且 Grace Drop 異常
  3. net_impact_hkd < 0（即 平台從這位司機身上賺到的佣金，少過他造成的用戶損失；講人話即 平台實際蝕錢）— BA 衍生計算值，非 BQ 原生欄位；由 BQ 兩項數據合併計算：平台佣金收入（平台從完成單實收的佣金）− 用戶傷害估算（例如加成多付、流失、客服補償）

詳見 一、net_impact_hkd 說明

評分算法（管理層說明版）：

  • 主要信號：release_rate、reaccept_same_order_flag、surcharge_delta、net_impact_hkd
  • 輔助標記：grace_drop_ratio、driver_cancel_rate_30d、高價單接受率偏高 / 低價單棄單偏高
  • 控制變數：booking_type、pickup_region、hour_of_day、waypoint_straight_distance_km、driver_arrived_at
先確認是否存在訂單操縱行為，再用時段、區域、距離、到達時間判斷訂單本身是否較難派。 履約速度慢或距離長，只能作為難度控制變數，不能直接推論外掛。
